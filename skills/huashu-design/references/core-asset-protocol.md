# Core Asset Protocol

Local compact reference adapted from Huashu Design v2.0.

Use this before designing for a named brand, real product, app, website, or visual identity.

## Priority

Core assets beat abstract style words.

1. Verify the product, brand, version, release status, and key facts when they could have changed.
2. Collect or identify the concrete assets that make the design about the real thing:
   - logo or wordmark
   - product image or real app/screen screenshot
   - current UI screenshots for software products
   - real data shape or content examples when relevant
   - official color/type cues only after core visual assets are known
3. Prefer official sources first. If unavailable, use reliable public sources and say what is missing.
4. Do not invent screenshots, product imagery, logos, release status, or specs.

## Fallback

If no reliable core assets can be found quickly:

- state the gap plainly
- use honest placeholders
- produce direction options instead of pretending to have a final brand-accurate design

## Review

Before claiming a visual artifact is ready, check:

- Does the first screen clearly show the real product/brand/object?
- Are assets current enough for the task?
- Are placeholders labelled or visually honest?
- Did the design use product/logo/UI evidence, not only a generic palette?
