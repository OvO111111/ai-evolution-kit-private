# Admin Platform Execution Gate Lesson

The user explicitly corrected a recurring failure mode in the internal tools platform work: Codex knew the right product/design reasoning but did not execute it as a hard pre-code gate. The resulting UI became a runnable but ugly, poorly structured backend with duplicated actions, weak metrics, fake/static account management at first, and reference material used as surface inspiration instead of acceptance standards.

Durable lesson:
- For admin platforms, internal tools, dashboards, multi-app consoles, account/permission pages, and monitoring tools, do not start coding until reference inheritance, page task matrix, business metric gate, interaction gate, and visual gate are explicit.
- A referenced PRD/prototype/design system must become an acceptance standard, not decorative inspiration.
- "It runs" is not a valid completion standard for platform UI. Completion requires browser screenshot review for page hierarchy, primary action uniqueness, implemented controls, readable diagnostic fields, and real admin-product quality.
- If the UI starts accumulating patches after user criticism, stop patching and re-establish the product/page/design gate first.
