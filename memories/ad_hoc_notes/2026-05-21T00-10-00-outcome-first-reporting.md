# Outcome-First Reporting

Date: 2026-05-21

User criticism: a project report listed models, endpoints, compile results, screenshots, and changed files, but did not answer whether the user's actual UI/product requirement was satisfied. The report was activity-first instead of outcome-first.

Adopted rule:

- Final reports must start by answering the user's actual requirement.
- State whether the requirement is satisfied, partially satisfied, or blocked.
- State the user-visible result first.
- State remaining gaps before technical evidence.
- Technical details, file lists, endpoints, screenshots, compile results, and generated reports are evidence, not the headline.
- For UI/prototype work, visible UI/UX changes come before code and API details.
- For product/PRD/workflow work, decisions, scope, behavior, acceptance result, and unresolved product risks come before implementation notes.
- If the requirement is not actually met, say so and keep working unless blocked.
