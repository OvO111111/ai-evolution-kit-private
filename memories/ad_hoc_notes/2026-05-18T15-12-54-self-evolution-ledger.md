## Private Full Knowledge Vault Requirement

The user wants a separate local/private knowledge vault that can store all chat records, project materials, and sensitive information, including content excluded from the generic export bundle. The goal is recall completeness: avoid missing key context because material was considered too sensitive for export.

This does not change the generic export policy. It creates a separate target:

- **Generic export bundle:** portable, redacted, safe for other AI systems.
- **Private full vault:** local/private, can contain sensitive raw material, governed by access controls and retrieval boundaries.

Concrete first implementation location:

- `%USERPROFILE%\.codex\private\knowledge-vault\README.md`
- `%USERPROFILE%\.codex\private\knowledge-vault\config\SCHEMA.md`
- `%USERPROFILE%\.codex\private\knowledge-vault\index\index.md`
- `%USERPROFILE%\.codex\private\knowledge-vault\logs\operations.md`

Recommended architecture:

1. **Raw archive:** immutable source copies for chats, WeChat/Feishu context, project docs, files, meeting notes, screenshots, PDFs, code references, and metadata. Store source, timestamp, owner/project, sensitivity tier, and deletion status.
2. **Compiled wiki:** LLM-maintained Markdown pages for entities, projects, people, decisions, timelines, contradictions, and summaries. This is readable working knowledge, not the only source of truth.
3. **Retrieval/index layer:** full-text search plus optional BM25/vector/graph retrieval. Candidates include qmd, GBrain-style graph/search, or a custom local index.
4. **Policy layer:** per-source sensitivity, access scope, redaction rules, prompt-injection sanitization, audit logs, and "do not include in generic export" flags.
5. **Ingest/lint loop:** ingest new sources, update wiki, append log, detect stale facts, orphan pages, contradictions, and missing cross-links.

Principle: store broadly in the raw private vault, retrieve selectively into prompts. Completeness belongs in storage; selectivity belongs in context.

## Export note

Export goal: the user should not have to re-train another AI from scratch. Export portable working habits, durable preferences, skills, and evolution decisions; do not export business data, private chat contents, account state, secrets, customer/payment records, or large project files.

When exporting to another AI, include this ledger plus the referenced skill files under:
- `%USERPROFILE%\.codex\skills\absorb-lessons\`
- `%USERPROFILE%\.codex\skills\web-access\`
- `%USERPROFILE%\.codex\skills\agent-reach\`
- `%USERPROFILE%\.codex\skills\wechat-pay-product-design\`
- any `open-design-*`, `caveman*`, `pm-prd`, and `desktop-control` skills relevant to the target AI's tool surface.

Recommended export bundle:
- self-evolution source ledger and concise adoption strategies
- global operating preferences and durable collaboration style
- reusable skills and their references/scripts when they do not contain secrets
- skill quality rules, browser/web access lessons, WeChat article access workflow, and design/product skills
- tool capability notes that explain what was tested and what failed

Exclude from the generic export bundle:
- WeChat work-group message bodies, search results, local indexes, and group allowlists
- account login state, cookies, tokens, API keys, OAuth grants, and local credential files
- customer records, payment/order/refund data, complaint records, and merchant/business-private details
- large project files, generated artifacts, local database dumps, screenshots with private data, and raw rollout logs unless redacted

If a target AI needs sensitive context for a specific task, create a task-local, minimal, redacted packet instead of adding it to the general export bundle.
